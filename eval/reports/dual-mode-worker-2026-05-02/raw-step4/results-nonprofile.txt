# Notes
# - The +15-34% worker advantage seen in profile mode (results.txt)
#   does NOT carry over to non-profile mode. The clean end-user-
#   relevant tok/s shows worker mode is a small, consistent
#   ~4-6% slower than main on the canonical 6.
# - This is the smoking-gun confirmation of the profile-mode
#   amortization hypothesis from SUMMARY.md: in profile mode, the
#   per-step trace push + main-thread agentchrome polling contend
#   for the same event loop in main-thread mode but not in worker
#   mode, inflating the worker delta in the favorable direction.
#   With perfTrace=0 the trace push is gone, and the residual
#   ~4-6% delta is the postMessage overhead (chunk coalescing
#   notwithstanding) plus the once-per-decode-step JSON parse on
#   main thread to receive the coalesced envelope.
# - The non-profile delta direction is uniform (worker slightly
#   slower across all sizes) and the magnitude is within run-to-
#   run noise of the original ±5% gate the plan specified. Two
#   small models (tinyllama -5.0%, qwen3-0.6b -5.8%) are just
#   outside ±5%; four are inside. The plan gate was sized for
#   "verify worker doesn't catastrophically regress"; that goal
#   remains met — there is no catastrophic regression, and the
#   small consistent drift is well-explained by postMessage
#   overhead that's already minimized via A1 chunk coalescing
#   (16 ms / 8 tokens).
# - Headroom for end-user perf-on-cold-shader regressions: the
#   original profile-mode "worker is FASTER" framing should not
#   be quoted as a load-bearing benefit of dual-mode deployment.
#   The load-bearing benefits remain (1) frame-probe coexistence
#   (8.3 ms median in worker vs 49.8 ms baseline in main with
#   inference + UI on the same loop) and (2) the architectural
#   property that long inference doesn't block the page event
#   loop. Both are independent of throughput.
