## Embed-surface analysis

Walked `src/inference/model-inference.ts` `forwardSingle()` graph builder
(residual-stream loop body roughly lines 793-909, post-loop norm + lm_head
at lines 912-922). Cross-referenced
`~/Repos/llama.cpp/src/models/qwen3.cpp` per the
`llama-cpp-as-arch-truth-source.md` doctrine.

In webllm's graph, the per-layer residual loop terminates at line 909
(`cur = wasm.opAdd(ffnOut, attnResidual)`); `cur` then carries the raw
last-layer residual stream. Lines 912-916 apply the final RMSNorm
(`weights.norm`, plus optional `normBias`) producing `finalNorm`. Lines
917-919 perform the `lm_head` matmul (using `weights.output` if present,
otherwise tying to `weights.tokEmb`); optional `outputBias` is added at
line 920 and the resulting `logits` is the graph's terminal node
(`graphBuildForwardExpand` at line 922).

| # | Tap-point | Insertion graph node (`model-inference.ts` line) | Tensor shape | Output-norm applied | Notes |
|---|---|---|---|---|---|
| 1 | Pre-output-norm | `cur` after the residual-loop close (line 909, just before line 912's `opRmsNorm`) | `[E, N]` | no | Raw last-layer residual; closest to encoder-inference.ts:528-536 convention (it downloads `[E, N]` from `finalHidden` straight into `poolAndNormalize`). |
| 2 | Post-output-norm | `finalNorm` after lines 912-916 (after the `opRmsNorm` × `weights.norm` mul, plus optional `normBias` add at 916), but before line 917's `lm_head` matmul | `[E, N]` | yes | Matches llama.cpp qwen3.cpp's `res->t_embd` tap-point exactly. |
| 3 | Skip norm + lm_head | Same as #1, but framed as halting graph construction before line 912 rather than tapping mid-graph | `[E, N]` | no | Implementation-shape variant of #1; chosen only if we want to avoid materializing the norm op at all. |

**Recommendation:** Tap-point #2 (post-output-norm `finalNorm`).
Reasoning: `~/Repos/llama.cpp/src/models/qwen3.cpp:91-98` builds
`cur = inpL` after the residual loop, applies `build_norm(... output_norm
... LLM_NORM_RMS ...)`, tags the result `result_norm`, and then
**explicitly assigns `res->t_embd = cur` at line 98 — before the
`lm_head` matmul at line 101**. That `t_embd` slot is the canonical
embedding output for Qwen3 in llama.cpp; aligning the webllm tap-point
with `t_embd` means causal-LM-derived embedders (e.g.
Qwen3-Embedding-0.6B) match upstream pooling semantics and reference
vector sets generated by llama.cpp will be directly comparable in
Stage 2. The encoder-inference.ts `[E, N]` convention (line 528-536) is
preserved either way since both tap-points #1 and #2 are `[E, N]` —
#2 just adds the final RMSNorm that upstream applies.

**Architecture truth source consulted:**
`~/Repos/llama.cpp/src/models/qwen3.cpp:91-104` (post-loop
`cur = inpL`, `build_norm(... output_norm ...)`, `res->t_embd = cur`,
then `build_lora_mm(model.output, cur)` for `t_logits`).

**Query instruction-prefix (exact string from README's `get_detailed_instruct`):**

```python
def get_detailed_instruct(task_description: str, query: str) -> str:
    return f'Instruct: {task_description}\nQuery:{query}'
```

Concrete form (with the README's example task):

```
Instruct: Given a web search query, retrieve relevant passages that answer the query
Query:What is the capital of China?
```

Note the literal `\n` between the `Instruct:` line and the `Query:` line, and the absence of a space between `Query:` and the query text — both are verbatim from the README. Stage 2 must preserve them exactly.

**Document instruction-prefix:** none — documents are passed raw without any prefix.

**Divergences flagged:**
1. **L2 normalization is consumer-side.** The GGUF carries no flag indicating embeddings are pre-normalized; the Stage-2 webllm pipeline must apply L2 in JS after readback (matches encoder-inference.ts `poolAndNormalize` convention).
2. **Second end-of-turn token (`eot = 151645`).** The GGUF surfaces `<|im_end|>` (151645) alongside `<|endoftext|>` (151643). Embedding mode should not stop on either — sequences run to caller-controlled length. Documenting because the chat-mode regression-lessons in `CLAUDE.md` already flag this dual-EOT pattern for Qwen3 generation.
3. **Vocab-size key not in GGUF metadata.** HF reports 151669; the GGUF probe surfaced the tokens table as `<object>` rather than a scalar count. If Stage 2 needs the vocab size programmatically, it must be derived from the tokens table length, not read from a metadata key.
