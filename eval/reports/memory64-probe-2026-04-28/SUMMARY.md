## 1. Headline

- ✅ **Phase 1 PASS.** ASYNCIFY × MEMORY64 round-trip works on current
  Emscripten + Chrome. The single load-bearing risk axis identified in
  spec §4.1 is **retired**.
- ❌ **Phase 2 FAIL.** BigInt pointer ABI is broken for stdlib
  `_malloc` / `_free` exports (returns `Number` instead of `BigInt`),
  while custom-bridge exports (`_tensor_new_1d`, etc.) correctly return
  `BigInt`. This is a targeted Emscripten config gap, not a fundamental
  blocker.
- ⚠️ **Phase 3 result is invalid.** The cap probe loop bailed at
  iteration 0 because `_malloc(1 GiB)` returned a JS `Number` (per the
  Phase 2 finding); the loop's type guard treated this as malloc
  failure. We have **no measured cap value** from this run.
- ✅ **Phase 4 PASS.** Re-init/shutdown after the (no-op) heap pressure
  works. Runtime is stable under MEMORY64.

## 2. Decision-rule branch (spec §5.1)

> **"Phase 1 passes, Phase 2 fails. Narrower follow-up: investigate the
> specific ABI failure before committing more surface."**

This is exactly the branch the probe landed on. The cap measurement
(Phase 3) cannot be trusted until the BigInt ABI gap is fixed; doing so
is a small targeted change (not a full TS-bridge migration).

## 3. Findings in detail

### 3.1 Why custom exports work but `_malloc` doesn't

The harness log shows the asymmetry directly:

```
_tensor_new_1d → typeof=bigint value=0xac65a0      ← OK
_malloc(16n)   → typeof=number value=0xac6548      ← BROKEN (returned Number)
```

Both functions are listed in `EXPORTED_FUNCTIONS`. Both return `void *`
in C. Under `-sMEMORY64=1 -sWASM_BIGINT=1`, both should marshal as
BigInt to JS. The difference is that custom exports get their wrapper
generated from header signatures the linker sees during the build,
whereas `_malloc` is a stdlib primitive whose JS shim is generated by
Emscripten directly. The shim apparently doesn't honor MEMORY64 + BIGINT
for stdlib pointers in this Emscripten release.

**Likely fixes** (to investigate in the follow-up sub-probe, _not_ in
this report):

1. Pass `--js-library` overrides for `_malloc`/`_free`.
2. Emit a thin C wrapper (`bridge_malloc(size_t)` / `bridge_free(void*)`)
   that the build already declares with full signatures, and use that
   from JS instead of the stdlib name.
3. Try a newer Emscripten point release; this may already be fixed.

Approach (2) is a few-line change and the fastest probe path.

### 3.2 What we _do_ know about the cap

Nothing useful. Iter-0 reported `_malloc returned 0` but the value was a
JS `Number` from a 32-bit-truncated pointer; we cannot tell if the WASM
allocator actually returned 0 (cap = 0) or returned a high pointer
value that got mangled to 0 in transit. The cap measurement must be
re-done after fix 3.1.

### 3.3 ASYNCIFY × MEMORY64 compatibility (load-bearing risk)

Spec §4.1 called this "**the load-bearing risk axis**" — if ASYNCIFY
were incompatible with MEMORY64, the entire lever would be closed
without a measurement-driven path back. **It is not.** Phase 1 took
1.4 ms wall time for `_webgpu_init` (which routes through async device
acquisition), and Phase 4 successfully re-ran init+shutdown after Phase
3 exercised the heap. This is the most valuable single bit produced by
the probe.

## 8. Follow-up

Recommended next step (out-of-scope for this probe, scoped per spec
§6): a small follow-up sub-probe that adds C-wrapped `bridge_malloc` /
`bridge_free` exports and re-runs only Phase 2 + Phase 3. If that
returns a credible cap value (`> 0`), the original decision rule from
§5.1 applies to the new cap.

**Lever status:** **NOT closed.** Specifically:
- ASYNCIFY axis: cleared.
- BigInt ABI: targeted gap, fixable with a few-line bridge change.
- Cap measurement: deferred to the follow-up sub-probe.

The probe paid for itself: it surfaced the actual blocker (a config
gap, not an architectural incompatibility) at sub-day cost, on the same
day the spec was written.
